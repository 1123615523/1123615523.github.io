索引的作用

索引的出现是为了提高数据查询的效率

索引的数据结构

MySQL采用B+树存储索引，一个B+树有如下几个特征：

  1. 有k个子树的中间节点包含有k个元素（B树中是k-1个元素），每个元素不保存数据，只用来索引，所有数据都保存在叶子节点。
  2. 所有的叶子结点中包含了全部元素的信息，及指向含这些元素记录的指针，且叶子结点本身依关键字的大小自小而大顺序链接。
  3. 所有的中间节点元素都同时存在于子节点，在子节点元素中是最大（或最小）元素。

根据图可以得出结论：

  1. 根节点元素8是子节点2,5,8 的最大元素，也是叶子节点6,8 的最大元素；
  2. 根节点元素15是子节点11,15 的最大元素，也是叶子节点13,15 的最大元素；
  3. 根节点的最大元素也就是整个B+树的最大元素，以后无论插入删除多少元素，始终要保持最大的元素在根节点当中。
  4. 由于父节点的元素都出现在子节点中，因此所有的叶子节点包含了全部元素信息，并且每一个叶子节点都带有指向下一个节点的指针，形成了一个有序链表。

根据叶子节点的内容，索引类型分为主键索引和非主键索引，根据两种索引查询有什么不同？

主键索引的叶子节点存的是整行数据。在 InnoDB 里，主键索引也被称为聚簇索引（clustered index）。

非主键索引的叶子节点内容是主键的值。在 InnoDB 里，非主键索引也被称为二级索引（secondary index）。

如果语句是 select * from T where ID=500，即主键查询方式，则只需要搜索 ID 这棵 B+ 树；

如果语句是 select * from T where k=5，即普通索引查询方式，则需要先搜索 k 索引树，得到 ID 的值为 500，再到 ID 索引树搜索一次。这个过程称为回表。

也就是说，基于非主键索引的查询需要多扫描一棵索引树，尽量减少回表操作可以提升查询速度。

索引维护

B+ 树为了维护索引有序性，在插入新值的时候需要做必要的维护。以上面这个图为例，如果插入新的行 ID 值为 700，则只需要在 R5 的记录后面插入一个新记录。如果新插入的 ID 值为 400，就相对麻烦了，需要逻辑上挪动后面的数据，空出位置。

如果 R5 所在的数据页已经满了，根据 B+ 树的算法，这时候需要申请一个新的数据页，然后挪动部分数据过去。这个过程称为页分裂。

页分裂操作还影响数据页的利用率。原本放在一个页的数据，现在分到两个页中，整体空间利用率降低大约 50%。

分裂就有合并。当相邻两个页由于删除了数据，利用率很低之后，会将数据页做合并。合并的过程，可以认为是分裂过程的逆过程。

覆盖索引

对一张业务数据库表创建索引，如果select后面的查询的参数列表都可以通过这个索引的树中获取到，这种情况就利用到了覆盖索引，MySQL的explain执行计划分析中的extra的using index会标明当前SQL是否利用到了覆盖索引。

如果没有利用到覆盖索引，需要回到主键索引（聚簇索引）中再做一次树搜索，这个过程叫做回表。

由于覆盖索引可以减少树的搜索次数，显著提升查询性能，所以使用覆盖索引是一个常用的性能优化手段。

最左前缀原则

MySQL创建根据字段顺序创建联合索引如（a，b，c），在进行时会从最左侧（a）这个字段开始，如果不遵从最左原则那么会全表扫描搜索数据。

最左优先，以最左边的为起点任何连续的索引都能匹配上。同时遇到范围查询(>、<、between、like)就会停止匹配。

索引查询失效场景

  • 联合索引不符合最左匹配原则
  • 使用了select *
  • 索引列参与了运算
  • 索引列使用了函数
  • like左边包含%
  • 类型隐式转换
  • 进行索引的列对比
  • not in和not exist

总结

MySQL采用B+Tree维护索引结构和存储数据，B+Tree非叶子节点不存储数据只存储数据指针，叶子节点依据关键字从小到大依次排序连接，B+Tree相比其他数据结构（红黑树，AVL树，B-Tree）矮且胖，那么InnoDB在进行树搜索数据极大的减少了IO次数，其他树结构层级不稳定树越高从磁盘中查找数据耗费时间就越久。

MySQL根据联合索引作为查询需要遵循最左前缀原则，索引检索从最左侧开始，否则将会全表查询，MySQL从5.6开始引入索引下推优化索引查询数据，减少回表次数提升效率。

索引的维护会重新排列B+Tree结构，效率会受影响，主要有页分裂和页合并的过程，针对不同业务表设计合理索引结构可以提升MySQL的写入速度。

索引常见失效场景需要注意，可以执行explain查看SQL执行计划分析结果。
